# Implement All

Autonomously implement and test all pending tasks in sequence. This is the "build the whole thing" command.

## Important Guidelines

- **Fully autonomous execution** — the AI implements every pending task, runs tests, and moves to the next without pausing for confirmation between tasks.
- **Pause only on failure** — if a task fails after retries, pause and ask the user what to do before continuing.
- **Always use AskUserQuestion tool** when asking the user anything.
- This command orchestrates `/implement-task` and `/test-task` logic internally — it does NOT call those commands, it follows their protocols inline.

## Process

### Step 1: Load full context

Read all of the following:

- `spec-bot/state.json` — tasks, features, testResults, phase
- `spec-bot/product/tech-stack.md` — languages, frameworks, test runner
- `spec-bot/product/mission.md` — product goals
- `spec-bot/product/roadmap.md` — scope
- `spec-bot/specs/` — any spec files

If `state.tasks` is empty:

```
No tasks found. Run /store-task to generate tasks first.
```

### Step 2: Check prerequisites

1. **Are there pending tasks?** — filter tasks with `status: "pending"`. If none, report status and stop:

```
No pending tasks. [X] tasks done, [Y] verified, [Z] failed.
Run /list-tasks to see the full status or /validate-project for final checks.
```

2. **Does a project scaffold exist?** — check for dependency manifests, source directories, config files. If the project root is essentially empty (no source files, no dependency files), stop:

```
No project structure detected. Run /scaffold-project first to set up the codebase, then re-run /implement-all.
```

### Step 3: Plan execution order

Order tasks for implementation:

1. **Group by feature** — implement all tasks for one feature before moving to the next.
2. **Feature priority** — features that are already `"in-progress"` come first (finish what you started). Then `"pending"` features in id order.
3. **Task order within a feature** — by task id (t1 before t2). Tasks within a feature are assumed to be in a logical build order as generated by `/store-task`.

Present the execution plan and use AskUserQuestion:

```
Ready to implement [N] pending tasks across [M] features:

  Feature: [f1] [Name] — [count] tasks
    t1 — [Name]
    t2 — [Name]

  Feature: [f2] [Name] — [count] tasks
    t3 — [Name]
    t4 — [Name]

Estimated: [N] implementation + test cycles.

Options:
1. Start (implement all, pause only on failure)
2. Start from a specific task (e.g. "start from t3")
3. Implement only one feature (e.g. "only f2")
4. Cancel
```

### Step 4: Execute the loop

For each task in the execution order, follow this cycle:

#### 4a. Scan and plan

- Read existing codebase (especially files created by previous tasks in this run).
- Plan which files to create/modify.
- Output a brief summary (no confirmation needed):

```
--- Task [X]/[N]: t1 — [Name] ---
Creating: [file list]
Modifying: [file list]
```

#### 4b. Set in-progress

Update `state.json`: task status → `"in-progress"`, feature status → `"in-progress"` (if not already).

#### 4c. Implement (tests first, then code)

Follow the same protocol as `/implement-task` Steps 5-6:

1. Install any new dependencies needed.
2. Write tests from `testCases`.
3. Write the implementation.
4. Run tests.
5. If tests fail, analyze and fix (up to 3 retries).

#### 4d. Handle failure

If a task fails after 3 retries:

- Set task status to `"blocked"`
- Use AskUserQuestion:

```
Task t[X] — [Name] is blocked after 3 fix attempts.

Failing: [brief error]

Completed so far: [Y]/[N] tasks

Options:
1. Show full error — I'll investigate
2. Skip this task, continue with the rest
3. Stop here — I'll resume later with /implement-all
```

- If **1**: show output, wait for user guidance, attempt fix, then continue.
- If **2**: set status to `"blocked"`, continue with next task.
- If **3**: save state and stop.

#### 4e. Mark done and continue

- Set task status → `"done"`
- If all tasks for the feature are done, set feature status → `"done"`
- Run a quick test to verify nothing broke from previous tasks (run full test suite every 3 tasks, not every single task — for speed).
- Move to next task.

### Step 5: Final test pass

After all tasks are implemented:

1. Run the **full test suite** (not just individual task tests).
2. If anything fails that was previously passing, identify which task's implementation broke it and fix it.
3. Record a `_full_suite` entry in `testResults`.

### Step 6: Update state

Update `spec-bot/state.json`:

- All implemented tasks should be `"done"` or `"verified"` or `"blocked"`
- All features with all tasks done should be `"done"`
- Set `phase` to `"qa"` (ready for validation)
- Update `updatedAt`

Update `spec-bot/context.md` with final summary.

### Step 7: Report

Use AskUserQuestion:

```
Implementation complete.

Results:
  [X] tasks implemented and passing
  [Y] tasks blocked (need attention)
  [Z] features complete

Full test suite: [PASSED/FAILED]

Next steps:
  - Run /validate-project for end-to-end validation
  - Run /test-task all to do a thorough test review
  - Run /list-tasks to see the full status
```
